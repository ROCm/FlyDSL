# FlyDSL CI Performance Dashboard

A static dashboard for FlyDSL CI benchmarks, served at
**<https://rocm.github.io/FlyDSL/ci-dashboard/>**. Four views:

| Tab | What it shows |
|---|---|
| **Health** | The landing page. A single count of *real* regressions on main, then those kernels with op / shape / arch / ќФ% and a sparkline. "Real" = a drop beyond the kernel's run-to-run noise (or the вИТ3% gate when samples are sparse). |
| **PR Check** | Pick a PR вЖТ does its latest run slow any kernel vs main, with real-vs-watch counts. The merge-gating view. |
| **Trends** | Per-kernel TB/s ¬Ј TFLOPS across commits, one line per runner, with a main mean ¬± 2ѕГ noise band (single-arch view). |
| **CI Board** | Recent PRs/runs with per-runner pass/fail/running chips and each run's worst ќФ vs main. Live from the public Actions API, snapshot fallback. |

No build step, no framework: plain HTML/CSS/JS + a vendored `vendor/chart.umd.min.js`.

## How it fits together

```
flydsl.yaml "Fly DSL test" run  вФАвФАlogsвФАвФАвЦЇ  ci-dashboard-ingest.yml
   (3 GPU runners, per PR/main)               parse_bench.py вЖТ ingest.py
                                                     вФВ commit
                                                     вЦЉ
                            branch  ci-dashboard-data   history.json ¬Ј runs.json
                                                     вФВ fetched at runtime (raw.githubusercontent)
docs.yml  вФАвФАcopies ci-dashboard/ into the docs Pages artifactвФАвФАвЦЇ  /ci-dashboard  (the page)
                                                     + live PR status via the public Actions API
```

The **page** ships inside the existing docs Pages deploy (`docs.yml` copies this
folder into `docs/_build/html/ci-dashboard`) вАФ no GitHub Pages setting change.
The **data** is regenerated by `ci-dashboard-ingest.yml` (cron + after each
benchmark run) onto the `ci-dashboard-data` branch, which the page fetches live.

## Pieces

- `index.html`, `styles.css`, `app.js` вАФ the dashboard (data wiring in `app.js`, config at the top).
- `ingest/parse_bench.py` вАФ parses `run_benchmark.sh` output + the `=== Benchmark: current vs вА¶ ===`
  comparison blocks into normalized records. Stdlib-only; runs in CI and locally.
- `ingest/ingest.py` вАФ lists recent runs via `gh`, pulls the 3 benchmark jobs' logs, merges history,
  snapshots run status.
- `data/` вАФ seed `history.json` / `runs.json` (first-paint fallback) and `schema.md` (the contract).
- `../.github/workflows/ci-dashboard-ingest.yml` вАФ the ingestion workflow.

## Local development

```bash
cd ci-dashboard && python3 -m http.server 8137
# open http://127.0.0.1:8137/  (loads the bundled seed; live board hits the public API)
```

To refresh the seed / data branch from real runs:

```bash
python3 ci-dashboard/ingest/ingest.py --repo ROCm/FlyDSL --max-runs 40 --out-dir /tmp/out
```

Tabs are deep-linkable: `#health`, `#prcheck`, `#trends`, `#board`. Keys `1`вАУ`4` switch tabs, `r` reloads.

## Configuration & forks

All wiring lives in the `CFG` object at the top of `app.js` (`repo`, `dataBranch`,
`api`, regression thresholds). A fork that wants its own dashboard points `CFG.repo`
/ `CFG.dataBranch` at its org and runs the ingest workflow there.

The live CI board uses the **unauthenticated** public Actions API (60 requests/hour
per IP). That is ample for the default 90 s refresh and the вЙ§6 requests per load; if
the limit is hit the board degrades gracefully to the last `runs.json` snapshot. The
benchmark numbers never depend on the live API вАФ they come from the data branch.
